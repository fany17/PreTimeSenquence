# Contributing

## 修改类型

提交应尽量只包含一种目的：

- `docs:` 文档与研究规范；
- `data:` 数据获取、校验与 schema；
- `target:` Ground Truth；
- `model:` 模型与校准；
- `backtest:` 成交、账户和风险模拟；
- `test:` 自动化测试；
- `refactor:` 不改变研究口径的结构整理。

## 文档规范

- 根目录只保留项目入口、状态和协作规则；
- 规范性设计文档使用大写英文文件名；
- 历史实验文档保留原始结果，不在后续修改中重写旧结论；
- 新实验从 `docs/EXPERIMENT_TEMPLATE.md` 复制；
- 每个结论必须指向对应的数据、命令或输出；
- 使用“已实现 / 已验证 / 计划中 / 未验证”等明确状态词。

## 代码规范

- Python 模块使用 `snake_case`；
- 配置、输入和输出使用类型标注；
- 数据处理保持确定性并显式处理时区；
- 禁止静默 fallback 到另一套模型或策略；模型缺失应立即失败；
- 不在模块内部硬编码币种、日期、杠杆或手续费；
- 一个 `StrategySpec` 必须贯穿 target、训练、回测和 paper trading。

## 测试要求

最少覆盖：

1. 数据排序、重复、缺口和时区；
2. bar t 特征不得读取 t+1；
3. entry 使用下一可成交时点；
4. 同一 bar 同时触及 TP/SL；
5. purge/embargo 边界；
6. TP/SL 后持仓恢复 flat；
7. fold 结束强制退出；
8. 手续费、滑点、资金费率和账户权益；
9. 固定随机种子下实验可复现。

## 数据与安全

以下内容不得进入普通 Git 历史：

- API key、secret、账户与订单信息；
- 大型原始数据和衍生数据；
- 训练模型；
- HTML 图、日志和临时输出。

仓库只保存数据 manifest、获取配置、schema、checksum 和小型测试 fixture。

## 合并前检查

- [ ] 修改范围与任务一致；
- [ ] 文档状态与实现一致；
- [ ] 无未来信息泄漏；
- [ ] 相关测试通过；
- [ ] 新实验使用标准模板；
- [ ] 未提交密钥、数据、模型和输出；
- [ ] 对失败结果和限制进行了记录。
